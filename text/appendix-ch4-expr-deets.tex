\subsection{Common Hyperparameters Reference}
\label{appsec:4:hyperparams}

    The following parameters were used by all algorithms considered in section \todo{ref gridworlds results}:
    \begin{itemize}
        \item $\mctsmode$ - a \thtspp\ewe parameter that specifies whether the selection phase should end when a new decision node is added to the tree or not;
        \item $\graphmode$ - a \thtspp\ewe parameter that specifies whether a transposition table should be used to store and re-use the decision nodes of the search tree when they are reached again via another path;
        \item $\normalisedq$ - if Q-values should be normalised before being used in the search policy;
        \item $\initv$ - the heuristic value function;
        \item $\initq$ - the heuristic Q-value function;
        \item $\Nv$ - the number of pseudo-visits assigned to $\initv$ in backups;
        \item $\Nq$ - the number of pseudo-visits assigned to $\initq$  to the Q-value estimate;
    \end{itemize}

    The following parameters are used by specific algorithms:
    \begin{itemize}
        \item $\epsilon$ - the (uniform) exploration parameter used by algorithms using a Boltzmann search policy, e.g. MENTS, BTS;
        \item $\alpha$ - the temperature parameter for algorithms using a maximum entropy objective, e.g. MENTS, RENTS, TENTS;
        \item $\alpha(x)$ - the temperature schedule for Boltzmann distributions in BTS and DENTS;
        \item $\beta(x)$ - the entropy weighting function for DENTS;
        \item $b$ - the bias term used in UCB style policies, e.g. UCT, H-MCTS;
        \item $\uctbudget$ - \todo{check if its seq halving into uct or uct into seq halving}, the budget for running UCT before swtiching to Sequential Halving in H-MCTS;
    \end{itemize}

    \bd{To be compatible with hyperparameter optimisation, the functional form of } $\alpha$ and $\beta$ \bd{need to be specified in a parameterised form.} For the experiments in Section \todo{ref gridworlds results}, the following forms were used:
    \begin{align}
        \alpha(x) &= \frac{\alpha_0}{\sqrt{1+\alphadecay x}} \\
        \beta(x) &= \begin{cases}
            \beta_c - \frac{\beta_c}{\betathresh}x & \text{if } x \leq \betathresh, \\
            0 & \text{otherwise.}
        \end{cases}
    \end{align}

    These functions are chosen so that the entropy weighting becomes negligible, and more specifically so that $\frac{\beta(x)}{\alpha(x)}$ tends to zero as $x$ tends to infinity.


\subsection{Integration of Common Hyperparameters into Prior Algorithms}
\label{appsec:4:uct_general}

    Consider an algorithm with value estimates $\bar{V}(s)$ and $\bar{Q}(s,a)$. To use the $\Nq$ and $\normalisedq$ parameters, the biased Q-values $\Qnorm(s,a)$ used in the search policy are given by:
    %
    \begin{align}
        \Qbias(s,a) &= \frac{N(s,a) \bar{Q}(s,a) +  \Nq \Qinit(s,a)}{N(s,a)+\Nq}, \\
        \Qnorm(s,a) &= 
        \begin{cases}
            \frac{\Qbias(s,a)-\min_{a\in\cl{A}}\Qbias(s,a)}{\max_{a\in\cl{A}}\Qbias(s,a)-\min_{a\in\cl{A}}\Qbias(s,a)} & \text{if } \normalisedq, \\
            \Qbias(s,a) & \text{otherwise.}
        \end{cases}
    \end{align}

    All visit counts and value estimates are initialised to zero: $N(s)=0$, $N(s,a)=0$, $\bar{V}(s)=0$ and $\bar{Q}(s,a)=0$. While MCTS algorithms are often presented as initialising new leaf nodes directly with a visit and the heuristic value \todo{ref UCT init}, this biasing term can instead be encorporated into the backups instead, allowing the algorithm to be run with an without the $\mctsmode$ parameter.
    
    Given a trajectory $\tau=(s_0,a_0,r_0,\ldots,s_{h-1},a_{h-1},r_{h-1},s_h)$, the \textit{full backups} for the average returns $\bar{V}$ and $\bar{Q}$ including the biasing term for $t=h-1,\ldots,0$ are given by:
    \begin{align}
        \bar{Q}(s_t,a_t) &\leftarrow 
            R(s_t,a_t) + \sum_{s' \in \suc{s_t}{a_t}} \left( \frac{N(s'|s_t,a_t)}{N(s_t,a_t)} \bar{V}(s') \right), 
                    \label{eq:dents_q_backup} \\ 
        \bar{V}(s_t) &\leftarrow \frac{\Nv \Vinit(s_t)}{N(s_t)+\Nv} 
            + \sum_{a\in\cl{A}} \frac{N(s_t,a)}{N(s_t)+\Nv} \bar{Q}(s_t,a).
    \end{align}

    \bd{In the main thesis the visit counts } $N(s'|s,a)$ \bd{are breifly mentioned in Section XXX, but not used for the general presentation to follow prior works presentation.} When using $\graphmode$, all backups need to be of the \textit{full backups}, as discussed in Section \todo{ref} and argued by \todo{cite kata go}. 

    \bd{This is of particular importance for algorithms that used average returns that are typically computed using incremental backups. Note that the benefits of using} $\graphmode$ \bd{come when the unique state assumption does not hold. The correctness of using full backups when the unique state assumption does not hold is beyond the scope of this thesis} \todo{cite katago}.


\subsubsection{UCT Implementation with Full Backups}
\label{appsec:4:uct_full_backups}
    As a worked example, the version of UCT used in experiments that utilises all of the parameters from Section \ref{appsec:4:hyperparams} is given. The algorithm is specified minimally using the \thtspp\ewe schema given in Section \ref{sec:2-4-2-thts}.

    \paragraph{Value Estimates and Parameters.}
    
        \begin{itemize}
            \item $\Vuct(s)$ - The value estimate for state $s$;
            \item $\Quct(s,a)$ - The Q-value estimate for state $s$ and action $a$;
            \item $N(s)$ - The visit count for state $s$;
            \item $N(s,a)$ - The visit count for state $s$ and action $a$;
            \item $\normalisedq$ - A flag indicating whether the Q-values should be normalised when used in the search policy;
            \item $\Nvuct$ - The number of pseudo-visits to use for biasing the backups;
            \item $\Nquct$ - The number of pseudo-visits to use for biasing the Q-values;
            \item $\Vinit(s)$ - The heuristic value estimate for state $s$;
            \item $\Qinit(s,a)$ - The heuristic Q-value estimate for state $s$ and action $a$;
            \item $\buct$ - the bias parameter used in the UCT search policy;
        \end{itemize}

    \paragraph{Initialisation.}
        \begin{align}
            N(s) \leftarrow 0, \\
            N(s,a) \leftarrow 0, \\
            \Vuct(s) \leftarrow 0, \\
            \Quct(s,a) \leftarrow 0.
        \end{align}

    \paragraph{Search Policy.}
        Intermediate Q-values:
        \begin{align}
            \Quctbias(s,a) &= \frac{N(s,a) \Quct(s,a) +  \Nquct \Qinit(s,a)}{N(s,a)+\Nquct}, \\
            \Quctnorm(s,a) &= 
            \begin{cases}
                \frac{\Quctbias(s,a)-\min_{a\in\cl{A}}\Quctbias(s,a)}{\max_{a\in\cl{A}}\Quctbias(s,a)-\min_{a\in\cl{A}}\Quctbias(s,a)} & \text{if } \normalisedq, \\
                \Quctbias(s,a) & \text{otherwise.}
            \end{cases}
        \end{align}

        Search policy:
        \begin{align}
            \piuct(s) &= \argmax_{a\in\cl{A}} \Quctnorm(s,a) + \buct \sqrt{\frac{\log(N(s))}{N(s,a)}}, 
        \end{align}
        

    \paragraph{Backups.}
        Let $\tau_{:h}=(s_0,a_0,r_0,\ldots,s_{h-1},a_{h-1},r_{h-1},s_h)\sim\piuct$:
        \begin{align}
            \Quct(s_t,a_t) &\leftarrow 
                R(s_t,a_t) + \sum_{s' \in \suc{s_t}{a_t}} \left( \frac{N(s'|s_t,a_t)}{N(s_t,a_t)} \Vuct(s') \right), 
                        \label{eq:dents_q_backup} \\ 
            \Vuct(s_t) &\leftarrow \frac{\Nv \Vinit(s_t)}{N(s_t)+\Nv} 
                + \sum_{a\in\cl{A}} \frac{N(s_t,a)}{N(s_t)+\Nv} \Quct(s_t,a).
        \end{align}

    \paragraph{Recommendation Policy.}
        \begin{align}
            \psiuct(s)=\argmax_{a\in\cl{A}}\Quct(s,a),  \\
            \mvuct(s) = \argmax_{a\in\cl{A}} N(s,a). 
        \end{align}






\subsection{Hyperparameter Optimisation}
\label{appsec:4:hypersearch}

    The parameters shared between all algorithms were set manually to the following values for all environments:
    \begin{itemize}
        \item $\mctsmode=\text{False}$ - the selection phase continues until the planning horizon or a terminal state is reached;
        \item $\graphmode=\text{True}$ - transposition tables are used to store and re-use decision nodes when they are reached again via another path;
        \item $\normalisedq=\text{True}$ - Q-values are linearly normalised before being used in the search policy;
        \item $\Nv=0$ - the number of pseudo-visits assigned to $\initv$ in backups;
        \item $\Nq=1$ - the number of pseudo-visits assigned to $\initq$  to the Q-value estimate.
    \end{itemize}

    Using the principle of optimism in the face of uncertainty, the heuristic value functions were set to a trivial upper bound on the values that could be achieved in each environment. For the DeterministicGridWorld and FrozenLake environments, this meant the value functions were set to $\Vinit(s)=\Qinit(s,a)=1.0$ for all states $s$ and actions $a$. For the Sailing environments, because all rewards are negative, the value functions were set to $\Vinit(s)=\Qinit(s,a)=0.0$ for all states $s$ and actions $a$.

    To select the remaining hyperparameters for the experiments detailed in Section \todo{ref} the BayesOpt library \todo{cite} was used.     Recall from Appendix \ref{app:adapt_for_games} that we initialised values with the neural networks as $Q^{\text{init}}(s,a)=\log \tilde{\pi}(a|s)+B$ and $V^{\text{init}}(s)=\tilde{V}(s)$, where $B$ is a constant (adapted from Xiao \todo{fixcites}) %\etal \cite{xiao2019maximum}). 
    For these experiments we set a value of $B=\frac{-1}{|\mathcal{A}|}\sum_{a\in\mathcal{A}} \log\tilde{\pi}(a|s)$. Initialising the values in such a way tends to lead to the values of $Q^{\text{init}}(s,a)$ being in the range $[-20,20]$, to account for this, we scaled the results of the game to $100$ and $-100$, which means that any parameters selected are about $100$ times larger than they would be if we had not have used this scaling.
    
    In these experiments we used a recommendation policy that recommends the action that was sampled the most, i.e. $\psi(s)=\max_a N(s,a)$, as this tends to be more robust to any noise from neural network outputs.
    
    To select each parameter we ran a round robin tournament where we varied the value of one parameter. The agent that won the most games was used to select the parameter moving forward, and in the case of a tie we used the agent which won the most games. If the winning agent had the largest or smallest value, then we ran another tournament adjusting the values accordingly.
    
    We tuned all of these algorithms using the game of 9x9 Go with a komi of 6.5, and giving each algorithm 2.5 seconds per move. For our final results we used the same parameters on 19x19 Go with a komi of 7.5, giving each algorithm 5.0 seconds per move.
