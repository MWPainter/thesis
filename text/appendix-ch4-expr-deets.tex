\section{Hyperparameter Details for DeterministicGridWorld, FrozenLake and Sailing Experiments} 
\label{appsec:4:hps}

    This section is organised as follows:
    \begin{itemize}
        \item Section \ref{appsec:4:hyperparams} gives a list of shared and algorithm-specific hyperparameters used for the experiments;
        \item Section \ref{appsec:4:bias_term_integration} describes how the common hyperparameters are integrated into prior algorithms, using UCT as an example;
        \item Section \ref{appsec:4:hypersearch} gives details on the hyperparameter optimisation and the final hyperparameters that were selected.
    \end{itemize}


\subsection{Common Hyperparameters Reference}
\label{appsec:4:hyperparams}

    The following parameters were used by all algorithms considered in section \cite{sec:4-4-results}:
    \begin{itemize}
        \item $\mctsmode$ - a \thtspp\ewe parameter that specifies whether the selection phase should end when a new decision node is added to the tree or not;
        \item $\graphmode$ - a \thtspp\ewe parameter that specifies whether a transposition table should be used to store and re-use the decision nodes of the search tree when they are reached again via another path;
        \item $\normalisedq$ - if Q-values should be normalised before being used in the search policy;
        \item $\initv$ - the heuristic value function;
        \item $\initq$ - the heuristic Q-value function;
        \item $\Nv$ - the number of pseudo-visits assigned to $\initv$ in backups;
        \item $\Nq$ - the number of pseudo-visits assigned to $\initq$  to the Q-value estimate;
    \end{itemize}

    The following parameters are used by specific algorithms:
    \begin{itemize}
        \item $\epsilon$ - the (uniform) exploration parameter used by algorithms using a Boltzmann search policy, e.g. MENTS, BTS;
        \item $\alpha$ - the temperature parameter for algorithms using a maximum entropy objective, e.g. MENTS, RENTS, TENTS;
        \item $\alpha(x)$ - the temperature schedule for Boltzmann distributions in BTS and DENTS;
        \item $\beta(x)$ - the entropy weighting function for DENTS;
        \item $b$ - the bias term used in UCB style policies, e.g. UCT, H-MCTS;
        \item $\uctbudget$ - the search budget for UCT between each round of Sequential Halving in H-MCTS;
    \end{itemize}

    To be compatible with hyperparameter optimisation, the functional form of $\alpha$ and $\beta$ need to be specified using tunable scalar paramters. Hence for the experiments in Section \ref{sec:4-4-3-1-gridworld_results}, the following forms were used:
    \begin{align}
        \alpha(x) &= \frac{\alpha_0}{\sqrt{1+\alphadecay x}} \\
        \beta(x) &= \begin{cases}
            \beta_c - \frac{\beta_c}{\betathresh}x & \text{if } x \leq \betathresh, \\
            0 & \text{otherwise.}
        \end{cases}
    \end{align}

    These functions are chosen so that the entropy weighting becomes negligible, and more specifically so that $\frac{\beta(x)}{\alpha(x)}$ tends to zero as $x$ tends to infinity.


\subsection{Integration of Common Hyperparameters into Prior Algorithms}
\label{appsec:4:bias_term_integration}

    Consider an algorithm with value estimates $\bar{V}(s)$ and $\bar{Q}(s,a)$. To use the $\Nq$ and $\normalisedq$ parameters, the biased Q-values $\Qnorm(s,a)$ used in the search policy are given by:
    %
    \begin{align}
        \Qbias(s,a) &= \frac{N(s,a) \bar{Q}(s,a) +  \Nq \Qinit(s,a)}{N(s,a)+\Nq}, \\
        \Qnorm(s,a) &= 
        \begin{cases}
            \frac{\Qbias(s,a)-\min_{a\in\cl{A}}\Qbias(s,a)}{\max_{a\in\cl{A}}\Qbias(s,a)-\min_{a\in\cl{A}}\Qbias(s,a)} & \text{if } \normalisedq, \\
            \Qbias(s,a) & \text{otherwise.}
        \end{cases}
    \end{align}

    All visit counts and value estimates are initialised to zero: $N(s)=0$, $N(s,a)=0$, $\bar{V}(s)=0$ and $\bar{Q}(s,a)=0$. While MCTS algorithms are often presented as initialising new leaf nodes directly with a visit and the heuristic value (Section \ref{sec:2-4-2-uct}), this biasing term can instead be encorporated into the backups instead, allowing the algorithm to be run with an without the $\mctsmode$ parameter.
    
    Given a trajectory $\tau=(s_0,a_0,r_0,\ldots,s_{h-1},a_{h-1},r_{h-1},s_h)$, the \textit{full backups} for the average returns $\bar{V}$ and $\bar{Q}$ including the biasing term for $t=h-1,\ldots,0$ are given by:
    \begin{align}
        \bar{Q}(s_t,a_t) &\leftarrow 
            R(s_t,a_t) + \sum_{s' \in \suc{s_t}{a_t}} \left( \frac{N(s'|s_t,a_t)}{N(s_t,a_t)} \bar{V}(s') \right), 
                    \\ 
        \bar{V}(s_t) &\leftarrow \frac{\Nv \Vinit(s_t)}{N(s_t)+\Nv} 
            + \sum_{a\in\cl{A}} \frac{N(s_t,a)}{N(s_t)+\Nv} \bar{Q}(s_t,a).
    \end{align}

    In the main thesis the visit counts $N(s'|s,a)$ are breifly mentioned in Section \ref{sec:2-4-2-thts}, but not used for the general presentation to follow prior works presentation. When using $\graphmode$, all backups need to be of the \textit{full backups}, as discussed in Section \ref{sec:2-4-2-thts} and argued by \cite{katago}. 


\subsubsection{UCT Implementation with Full Backups}
\label{appsec:4:uct_full_backups}
    As a worked example, the version of UCT used in experiments that utilises all of the parameters from Section \ref{appsec:4:hyperparams} is given. The algorithm is specified minimally using the \thtspp\ewe schema given in Section \ref{sec:2-4-2-thts}.

    \paragraph{Value Estimates and Parameters.}
    
        \begin{itemize}
            \item $\Vuct(s)$ - The value estimate for state $s$;
            \item $\Quct(s,a)$ - The Q-value estimate for state $s$ and action $a$;
            \item $N(s)$ - The visit count for state $s$;
            \item $N(s,a)$ - The visit count for state $s$ and action $a$;
            \item $\normalisedq$ - A flag indicating whether the Q-values should be normalised when used in the search policy;
            \item $\Nvuct$ - The number of pseudo-visits to use for biasing the backups;
            \item $\Nquct$ - The number of pseudo-visits to use for biasing the Q-values;
            \item $\Vinit(s)$ - The heuristic value estimate for state $s$;
            \item $\Qinit(s,a)$ - The heuristic Q-value estimate for state $s$ and action $a$;
            \item $\buct$ - the bias parameter used in the UCT search policy;
        \end{itemize}

    \paragraph{Initialisation.}
        \begin{align}
            N(s) \leftarrow 0, \\
            N(s,a) \leftarrow 0, \\
            \Vuct(s) \leftarrow 0, \\
            \Quct(s,a) \leftarrow 0.
        \end{align}

    \paragraph{Search Policy.}
        Intermediate Q-values:
        \begin{align}
            \Quctbias(s,a) &= \frac{N(s,a) \Quct(s,a) +  \Nquct \Qinit(s,a)}{N(s,a)+\Nquct}, \\
            \Quctnorm(s,a) &= 
            \begin{cases}
                \frac{\Quctbias(s,a)-\min_{a\in\cl{A}}\Quctbias(s,a)}{\max_{a\in\cl{A}}\Quctbias(s,a)-\min_{a\in\cl{A}}\Quctbias(s,a)} & \text{if } \normalisedq, \\
                \Quctbias(s,a) & \text{otherwise.}
            \end{cases}
        \end{align}

        Search policy:
        \begin{align}
            \piuct(s) &= \argmax_{a\in\cl{A}} \Quctnorm(s,a) + \buct \sqrt{\frac{\log(N(s))}{N(s,a)}}, 
        \end{align}
        

    \paragraph{Backups.}
        Let $\tau_{:h}=(s_0,a_0,r_0,\ldots,s_{h-1},a_{h-1},r_{h-1},s_h)\sim\piuct$:
        \begin{align}
            \Quct(s_t,a_t) &\leftarrow 
                R(s_t,a_t) + \sum_{s' \in \suc{s_t}{a_t}} \left( \frac{N(s'|s_t,a_t)}{N(s_t,a_t)} \Vuct(s') \right), 
                        \\ 
            \Vuct(s_t) &\leftarrow \frac{\Nv \Vinit(s_t)}{N(s_t)+\Nv} 
                + \sum_{a\in\cl{A}} \frac{N(s_t,a)}{N(s_t)+\Nv} \Quct(s_t,a).
        \end{align}

    \paragraph{Recommendation Policy.}
        \begin{align}
            \psiuct(s)=\argmax_{a\in\cl{A}}\Quct(s,a),  \\
            \mvuct(s) = \argmax_{a\in\cl{A}} N(s,a). 
        \end{align}






\subsection{Hyperparameter Optimisation}
\label{appsec:4:hypersearch}

    The parameters shared between all algorithms were set manually to the following values for all environments:
    \begin{itemize}
        \item $\mctsmode=\text{False}$ - the selection phase continues until the planning horizon or a terminal state is reached;
        \item $\graphmode=\text{True}$ - transposition tables are used to store and re-use decision nodes when they are reached again via another path;
        \item $\normalisedq=\text{True}$ - Q-values are linearly normalised before being used in the search policy;
        \item $\Nv=0$ - the number of pseudo-visits assigned to $\initv$ in backups;
        \item $\Nq=1$ - the number of pseudo-visits assigned to $\initq$  to the Q-value estimate.
    \end{itemize}

    Using the principle of optimism in the face of uncertainty, the heuristic value functions were set to a trivial upper bound on the values that could be achieved in each environment. For the DeterministicGridWorld and FrozenLake environments, this meant the value functions were set to $\Vinit(s)=\Qinit(s,a)=1.0$ for all states $s$ and actions $a$. 
    
    For the Sailing environments, because all rewards are negative, the value functions were set to $\Vinit(s)=\Qinit(s,a)=0.0$ for all states $s$ and actions $a$. The remaining hyperparameters were selected using Bayesian optimisation via the BayesOpt library \cite{bayesopt}. Table \ref{table:gridworld_hps_space} gives the space searched over for each parameter, and if parameters were sampled using a linear or logarithmic scale.

\section{Additional Details for Go Experiments} 
\label{appsec:go}

    These experiments were run with \mctsmode\ewe turned on.

    Because running Bayesian optimisation at the scale of the Go experiments would be prohibitively expensive, a round-robin tournament was used to select the hyperparameters. This section gives the results of the round-robin tournaments.

    For PUCT, the parameters used were copied from the reported values for Kata Go and Alpha Go Zero \cite{katago,alpha_go_zero}. 

    This section is organised as follows:
    \begin{itemize}
        \item Section \ref{appsec:go_ar_ments} outlines the implementation of the AR-MENTS, AR-RENTS and AR-TENTS algorithms;
        \item Section \ref{appsec:round_robin} gives details on the round-robin tournament used to select the hyperparameters for Go experiments;
        \item Section \ref{appsec:go_ar_vs_bellman} gives results for matches between the Bellman and average return versions of each algorithm.
    \end{itemize}






\subsection{AR-MENTS, AR-RENTS and AR-TENTS}
\label{appsec:go_ar_ments}

    To implement AR-MENTS the AR-DENTS (Section \ref{app:4:algo-ref}) algorithm was used, but with $\beta(x)=\alpha(x)=\alpha_{\text{fix}}$. Implementing AR-RENTS and AR-TENTS is similar, replacing the used of Shannon entropy $\cl{H}$ in the backups by the corresponding relative entropy and Tsallis entropy functions. For AR-TENTS, the search policy uses the \textit{sparsemax} function used in TENTS instead of the softmax function. 





\subsection{Go Round Robin Hyperparameter Selection}
\label{appsec:round_robin}

    The following parameters were tuned for the BTS, DENTS, MENTS, RENTS and TENTS algorithms:
    \begin{itemize}
        \item $\epsilon$ - the exploration parameter used by algorithms using a Boltzmann search policy, e.g. MENTS, BTS;
        \item $\epsilon_{\tilde{\lambda}}$ - the weighting of the prior policy used by algorithms using a Boltzmann search policy, e.g. BTS;
        \item $\alpha$ - the temperature parameter for algorithms using a maximum entropy objective, e.g. MENTS, RENTS, TENTS;
        \item $\alpha_{\text{init}}$ - the initial temperature parameter for BTS and DENTS algorithms;
        \item $\beta_{\text{init}}$ - the initial entropy coefficient for DENTS algorithms. 
    \end{itemize}

    The temperature and entropy coefficient functions used in BTS and DENTS for Go experiments were of the form $\alpha(x)=\alpha_{\text{init}}/\sqrt{x}$ and $\beta(x)=\beta_{\text{init}}/\sqrt{x}$ respectively. For DENTS, only the entropy coefficient was tuned independently and used the parameters tuned for BTS.
    
    Parameters are tuned one at a time, with a round-robin tournament used to select the best value for each parameter, keeping the remaining parameters fixed. The parameter that won the most matches is selected, with ties broken by the agent that won the most games. If the winning agent had the largest or smallest value, then another tournament is run adjusting the values accordingly.

    These tuning tournaments were run using the game of 9x9 Go with a komi of 6.5, and giving each algorithm 2.5 seconds per move.
